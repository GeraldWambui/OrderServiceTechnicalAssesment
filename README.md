# Order Service

A comprehensive order management system built with FastAPI and React, featuring authentication, role-based access control, payment processing, and webhook integration.

## üöÄ Quick Start

### One-Command Setup (Docker)

```bash
# Clone the repository
git clone <repository-url>
cd order-service

# Start everything with Docker
docker-compose up -d

# Access the admin UI
open http://localhost/admin
```

### Manual Setup

```bash
# Install dependencies
pip install -r requirements.txt

# Run setup script
chmod +x setup.sh
./setup.sh

# Or start manually
uvicorn main:app --reload
```

## üèóÔ∏è Architecture

### Core Components

- **FastAPI Backend**: RESTful API with automatic OpenAPI documentation
- **React Admin UI**: Modern single-page application for order management
- **PostgreSQL**: Primary database with optimistic locking
- **Redis**: Caching layer for improved performance
- **Nginx**: Reverse proxy and static file serving

### Key Features

#### Part A - Secure Backend Service
- ‚úÖ **JWT Authentication** with HS256 signing
- ‚úÖ **Role-Based Access Control** (ADMIN, USER)
- ‚úÖ **Idempotent Order Creation** with client tokens
- ‚úÖ **Optimistic Locking** for concurrent updates
- ‚úÖ **Input Validation** with detailed error messages
- ‚úÖ **Database Migrations** with SQLAlchemy
- ‚úÖ **Comprehensive Tests** (16 test cases)
- ‚úÖ **Request Logging & Metrics** with Prometheus-style counters
- ‚úÖ **Rate Limiting** (10 requests/minute for order creation)

#### Part B - External Integration
- ‚úÖ **Payment Webhook Integration** with HMAC signature verification
- ‚úÖ **Idempotent Webhook Handling** safe for replays
- ‚úÖ **Exponential Backoff Retry** (3 attempts with exponential delay)
- ‚úÖ **Response Caching** (30s TTL with cache invalidation)
- ‚úÖ **Dead Letter Queue** logging for failed retries

#### Part C - Admin UI
- ‚úÖ **JWT Authentication** with role verification
- ‚úÖ **Paginated Orders Table** with search functionality
- ‚úÖ **Real-time Status Updates** via PATCH requests
- ‚úÖ **Error Handling** with toast notifications
- ‚úÖ **Responsive Design** with Tailwind CSS

- **Message Queue**: Clustered RabbitMQ or AWS SQS for webhook processing

### Failure Scenario: Database Master Outage

#### **Scenario**
Primary PostgreSQL master crashes during peak traffic (5k req/s)

#### **Impact Without Mitigation**
- All write operations fail (order creation, status updates)
- Read operations continue via replicas
- Payment webhooks queue up, causing delays
- Estimated revenue loss: $50k/hour

#### **Mitigation Strategy**

**Immediate Response (0-5 minutes)**
```bash

# 2. Circuit breaker activates - API returns cached data
# 3. Queue webhook processing to prevent data loss
# 4. Alert on-call engineer
```
